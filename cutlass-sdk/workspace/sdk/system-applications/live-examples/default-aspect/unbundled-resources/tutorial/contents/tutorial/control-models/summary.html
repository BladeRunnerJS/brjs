Some presentation nodes are designed to be paired with View controls, and you can think of them as miniature presentation models in their own right. Such presentation nodes are called "control models", and they include:

caplin.presenter.node.Button:
Pairs with <button/> and <input type="submit"/> elements.
caplin.presenter.node.DisplayField:
Pairs with <span/> and <div/> elements.
caplin.presenter.node.Field:
Pairs with <input type="text"/> elements.
caplin.presenter.node.SelectionField:
Pairs with <select/> and <input type="radio"/> elements.
caplin.presenter.node.MultiSelectionField:
Pairs with <select multiple="multiple"/> and <input type="checkbox"/> elements.
caplin.presenter.node.DateField:
Pairs with <input type="text"/> and <input type="date"/> elements.
caplin.presenter.node.CalendarDateField:
Pairs with custom JavaScript calendar controls.

Creating separate models for specific controls has several advantages. Not least, it makes the intention of the presentation model clearer, and you can use common control-model templates to avoid repeating the same HTML, over and over again. As well as that though, standardising the property names makes the PresentationNode.properties() method more useful.

Although Presenter already has many commonly-used control models, it's a good idea to create your own tailored models for repeated HTML templates, bound to model instances.

It's important to remember that a presentation node is a logical representation (a model), whereas the HTML is a specific implementation of how the field appears on screen. You can use the same logical presentation node with different view controls to completely change the on-screen appearance. For instance, you could swap a drop-down list for radio buttons, or replace <input> and <label> tags with a third-party library component, such as the Ext.js text control.
