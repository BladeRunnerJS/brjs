Sometimes, whether or not the value of a field is allowable will depend on the values of other fields or properties. For instance, you might have two fields (let's call them min and max) that set the minimum and maximum values allowed for a third field, which we'll call amount. If the user changes any of those fields, so that amount is no longer higher than min and lower than max, you'll want to display an error.

Cross-Property Validation in Presenter

In Presenter, this concept of validating related values is called Cross-Property Validation. As you might recall from the tutorial section, a Field in Presenter is a collection of Property instances (value, visible, and so on), and validators are attached to properties. When you attach a validator to a field's value property, any validation errors propagate to the Field itself, setting the values of its hasError and failureMessage properties appropriately.

To perform Cross-Property Validation, first you have to create a class that implements the caplin.presenter.validator.CrossPropertyValidator interface. However, unlike a normal caplin.core.Validator, you don't attach an instance of a Cross-Property Validator directly to every property. Instead, use the bindValidator() method on the global singleton caplin.presenter.validator.CrossValidationPropertyBinder, which manages and binds Cross-Property Validators. This example will show you how this is done, with comments in the code itself to help clarify the details.

Some additional points:
  • When writing a new Cross-Property Validator, you should specify exactly what mappings mProperties is expected to contain, both in your documentation and to other developers, since it will differ for each new type of validator. You should also implement the validate() method to expect some properties to be missing and to handle the case when they are.
  • After binding a Cross-Property Validator, the effect of the binding on any caplin.presenter.property.EditableProperty is the same as if a normal validator was added to it with its addValidator() method. This means that the order in which validators are run is respected, as with normal validators.

In this example, when a cross-validation error occurs, all the properties are affected. Errors are then styled using the following stylesheet: 

.error{
    border:1px solid red !important;
}

.error_msg{
    color:red !important;
}
